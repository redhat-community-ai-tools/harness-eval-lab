# Security Rule Benchmarks

**Status:** future
**Created:** 2026-06-15

## Problem

harness-eval-lab has 22 security rules covering prompt injection, credential exposure, data exfiltration, obfuscation, reverse shells, MCP poisoning, and more. But there's no way to measure how well these rules actually perform. We don't know the detection rate against real-world attack patterns, the false positive rate on benign setups, or how the rules compare to other tools.

Without benchmarks:
- New rules ship without validated detection rates.
- Regressions in pattern matching go unnoticed.
- Users can't assess whether the security layer is thorough enough for their threat model.
- Claims about security coverage ("17 injection patterns") are feature-counts, not performance evidence.

## Proposal

Build a benchmark suite that measures security rule performance against a curated corpus of known-malicious and known-benign setups.

### Components

1. **Malicious corpus**: A collection of agent setup files containing real-world attack patterns. Each file is labeled with the expected findings (which rules should trigger, which lines, which severity). Sources:
   - Public prompt injection datasets (e.g., Garak, PromptBench, HackAPrompt)
   - Real incidents from blog posts, CVE reports, and security research
   - Synthetically generated adversarial setups (LLM-generated evasion attempts)
   - Community contributions (red team exercises)

2. **Benign corpus**: A collection of legitimate agent setups that should produce zero security findings. Sources:
   - Open-source Claude Code setups from GitHub
   - Setups generated by following official best practices
   - Setups with security-adjacent content that should NOT trigger (documentation about injection, code examples in fenced blocks, quoted references)

3. **Benchmark runner**: A script that runs all security rules against both corpora and produces a report with:
   - **True positive rate** (TPR): % of malicious patterns correctly flagged
   - **False positive rate** (FPR): % of benign content incorrectly flagged
   - **Per-rule breakdown**: TPR and FPR per individual rule
   - **Per-pattern breakdown**: detection rate per attack pattern category
   - **Comparison across versions**: regression detection when rules change

4. **Benchmark report**: A generated document (markdown + JSON) showing results, published alongside releases.

### Directory structure

```
tests/benchmarks/
    security/
        corpus/
            malicious/          # Labeled malicious setups
                injection/      # Prompt injection variants
                exfiltration/   # Data exfiltration patterns
                credentials/    # Credential exposure
                obfuscation/    # Encoding/evasion tricks
                reverse_shell/  # Reverse shell patterns
            benign/             # Clean setups that should pass
                real_world/     # Real open-source setups
                documentation/  # Setups with security docs/examples
        labels.yaml             # Expected findings per file
        run_benchmark.py        # Benchmark runner
        report.md               # Generated results (gitignored, rebuilt in CI)
```

### Label format

```yaml
# labels.yaml
malicious/injection/ignore_previous.md:
  expected:
    - rule: security/no-prompt-injection
      line: 5
      label: ignore_instructions
    - rule: security/no-prompt-injection
      line: 12
      label: role_play

benign/documentation/security_guide.md:
  expected: []  # No findings expected
```

### Metrics

| Metric | Target | Formula |
|--------|--------|---------|
| Overall TPR | >= 90% | true_positives / (true_positives + false_negatives) |
| Overall FPR | <= 5% | false_positives / (false_positives + true_negatives) |
| Per-rule TPR | >= 80% | per-rule true_positives / per-rule expected_positives |
| Evasion resistance | tracked | % of obfuscated variants detected |

## User stories

### Story 1: Validate a new security rule

**Given** a developer adds a new security rule (e.g., `security/sudo-escalation`)
**When** they run the benchmark suite
**Then** they see the rule's TPR and FPR against the corpus, and can verify it meets the 80% TPR / 5% FPR thresholds before merging.

### Story 2: Track regression across releases

**Given** a maintainer is preparing a release
**When** CI runs the benchmark suite
**Then** the report shows per-rule TPR/FPR compared to the previous release, highlighting any regressions in detection capability.

### Story 3: Demonstrate security coverage to adopters

**Given** a potential user wants to know how thorough the security layer is
**When** they check the benchmark report in the repository
**Then** they see concrete detection rates per attack category, not just feature counts.

### Story 4: Red team the tool

**Given** a security researcher wants to test evasion techniques
**When** they add obfuscated variants to the malicious corpus and run the benchmark
**Then** they see which evasion techniques bypass detection, and can contribute improvements.

## Requirements

1. Benchmark corpus must include at least 50 malicious samples across all 6 attack categories (injection, exfiltration, credentials, obfuscation, reverse shells, MCP abuse).
2. Benign corpus must include at least 20 clean setups, including setups with security-related documentation that should not trigger false positives.
3. Each malicious sample must be labeled with expected rule matches (rule ID, line number, pattern label).
4. Benchmark runner must produce both human-readable (markdown) and machine-readable (JSON) output.
5. CI must run the benchmark on every PR that touches `inspection/rules/security/`.
6. Regression threshold: no rule's TPR may drop below 80% compared to the previous tagged release.
7. The benchmark report must be reproducible (same corpus + same rules = same results, no LLM involved).

## Success criteria

- Benchmark suite runs in under 30 seconds (no LLM calls, pure static analysis).
- Overall TPR >= 90% against the malicious corpus.
- Overall FPR <= 5% against the benign corpus.
- At least one community-contributed evasion variant is included in the malicious corpus.
- Benchmark report is published with each release.

## Open questions

1. **Corpus sourcing:** Should malicious samples be real-world (scraped from public incidents) or synthetic (LLM-generated)? Real-world samples are more credible but harder to collect. A mix is likely best.

2. **Evasion scoring:** Should evasion resistance be a separate metric or folded into per-rule TPR? Evasion variants test the same rules but with obfuscation layers, so they could be tracked as a sub-category.

3. **Semantic review benchmarking:** The LLM-based semantic security review (layer 3) is non-deterministic. Should it be benchmarked separately with averaged results, or excluded from the deterministic benchmark suite?

4. **Comparison with other tools:** Should the benchmark include runs of other security scanners (Semgrep, CodeQL) against the same corpus for comparison? This would be valuable for positioning but adds maintenance burden.
